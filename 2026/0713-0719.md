#### 自学
- Unicodeコードポイント: 世界中のあらゆる文字（絵文字含む）に対して、重複しないように1つずつ割り当てられた世界共通の整理番号のこと。あくまで文字のID。
- 比較演算子(`and` や `or` など)とビット演算子（`&` や `|` など）は書き方が異なるだけだと勘違いしていたことに気づけた。
- 比較演算子: 条件式が正しいか・間違っているかの真偽値(`True` or `False`)を扱う
- ビット演算子: 主に整数（0と1の並び）を扱う。数値そのものを加工・計算する目的で使用される
- また、比較演算子は効率重視なショートサーキットの考え方が適用されるが、ビット演算子は左の結果に関係なく右の式も必ず実行してビット計算する
- `string`モジュールを使って、アルファベット小文字の `a` から `z` を指定する方法を知った
- 当初 `range('a','z')` を使っていたが、そもそも数値しか使えないという前提が抜けていたことがわかった
- 名前付きタプルを使うことで辞書のようにキーを文字列などにすることができる。また、辞書と違って Immutable なので値が変更されない。
- 名前付きタプルもオブジェクトでありインスタンスである

```python
Employee = namedtuple('Employee', ('name', 'age', 'income'))
```

- 上記のようなクラスの書き方を知った。この行を実行した時点で、Pythonの内部では Employee という名前の新しいクラス（型）が動的に作られるらしい。通常の class Employee: と書いてクラスを作るのと同じような状態になるということ。
- Python では、数値リテラル（コードの中に直接書く数字）の中にアンダースコア（`_`）を挟んで記述することが認められているようだ。桁区切り文字（Visual Underscores）と呼ばれる機能で、可読性を高めるために使用することができる。例えば、0 が多い場合に数え間違えるリスクがあるので、この機能をつかうことでリスクを軽減できる（例: 12000000 = 12_000_000）
- 「30代」のような年代の表現方法

```python
# x: 人物インスタンス

# 自分の書き方
(30 <= employee.age <= 39)

# 模範解答の書き方
(employee.age // 10 == 3)
```
- SQLにおける `NULL` の比較はバグを生みやすいので注意が必要
- SQLの `NOT IN` をコンピュータが処理するとき、内部的には `NOT (A OR B OR C)` という形から、「どれでもない（Aではなく、かつBではなく、かつCではない）」という AND の形に変換される（ド・モルガンの法則）
- SQLでは、`NULL` との比較（id <> NULL）の結果は `True` にも `False` にもならず、`Unknown`（判定不能） になる

#### 業務
- `unique()` と `drop_duplicates()`の使い分け
- `unique()` : どんな種類があるかをリストで確認したいとき
- `drop_duplicates()` : ユニーク化した後も Series の形式を保ったまま別の処理をしたいとき
- データフレームとシリーズを混同しないこと
- `df[df['カラム']] ` のように内側と外側にブラケットがあるときの考え方を整理した 
- 内側の `df[]` で条件に合うかどうかの「True / Falseのチェックシート」を作り、外側の `df[]` で、そのチェックシートを元に、「Trueの行」を丸ごと抜き出しているイメージ 
- `df[]` の中に `df[]` が来る場合と `[]` が来るときの考え方を整理できた
- `[]` が1つのケースと2つのケースの違い

```python
# カッコが1つ ＝ Series になる 
df_kvi_list['競合店'] 

# カッコが2つ ＝ 1列だけの DataFrame になる 
df_kvi_list[['競合店']] 
```
